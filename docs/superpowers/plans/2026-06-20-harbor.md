# Harbor Container Registry — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deploy Harbor container registry via Flux HelmRelease with external CloudNativePG PostgreSQL.

**Architecture:** Create 8 manifests under `clusters/production/harbor/` — Namespace, SealedSecret (PG), CNPG Cluster, SealedSecret (admin), HelmRepository, HelmRelease, Ingress, Kustomization. Helm chart handles PVCs, Services, Deployments. CNPG operator handles PostgreSQL lifecycle. Traefik/cert-manager handle ingress and TLS.

**Tech Stack:** Kubernetes, Flux v2 (HelmRelease/HelmRepository), CloudNativePG, Harbor Helm chart (helm.goharbor.io), SealedSecrets, Traefik, cert-manager.

### Task 2: PostgreSQL — Secret Template + CNPG Cluster

**Files:**
- Create: `clusters/production/harbor/pg-secret.yaml` (plain template, not committed to git — used to generate sealed version)
- Create: `clusters/production/harbor/pg-secret-sealed.yaml` (output of `kubeseal`)
- Create: `clusters/production/harbor/pg-cluster.yaml`

**Note:** The sealed secret must be created by the user with `kubeseal` — it cannot be generated without cluster access. The plain `pg-secret.yaml` is a reference template (gitignored via `**/*secret.yaml`).

- [ ] **Step 1: Create plain PG secret template (reference only)**

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: harbor-pg-secret
  namespace: harbor
type: Opaque
stringData:
  username: harbor
  password: <generate-strong-password>
```

Write to `clusters/production/harbor/pg-secret.yaml`. **This file is already gitignored** (`**/*secret.yaml` in `.gitignore`), so it won't be committed.

- [ ] **Step 2: Create CNPG Cluster manifest**

```yaml
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: harbor-pg
  namespace: harbor
spec:
  instances: 1
  postgresql:
    parameters:
      max_connections: "50"
      shared_buffers: "128MB"
  bootstrap:
    initdb:
      database: harbor
      owner: harbor
      secret:
        name: harbor-pg-secret
  storage:
    size: 5Gi
  resources:
    requests:
      memory: "128Mi"
      cpu: "100m"
    limits:
      memory: "512Mi"
      cpu: "500m"
```

Write to `clusters/production/harbor/pg-cluster.yaml`.

- [ ] **Step 3: Seal the PG secret**

User runs:
```bash
cat clusters/production/harbor/pg-secret.yaml | kubeseal \
  --namespace harbor \
  --format yaml \
  > clusters/production/harbor/pg-secret-sealed.yaml
```

Verify the sealed secret was created:
```bash
cat clusters/production/harbor/pg-secret-sealed.yaml | head -5
# Expected: apiVersion: bitnami.com/v1alpha1, kind: SealedSecret
```

- [ ] **Step 4: Commit PG manifests**

```bash
git add clusters/production/harbor/pg-cluster.yaml clusters/production/harbor/pg-secret-sealed.yaml
git commit -m "feat(harbor): add CloudNativePG cluster (5Gi, single instance)"
```

**Files:**
- Create: `clusters/production/harbor/helm-repository.yaml`

- [ ] **Step 1: Create HelmRepository**


Write to `clusters/production/harbor/helm-repository.yaml`.

- [ ] **Step 2: Create HelmRelease**

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2beta2
kind: HelmRelease
metadata:
  name: harbor
  namespace: harbor
spec:
  interval: 10m
  chart:
    spec:
      chart: harbor
      sourceRef:
        kind: HelmRepository
        name: harbor
  values:
    expose:
      type: ingress
      tls:
        enabled: false
      ingress:
        hosts:
          core: harbor.malrusayni.com
        controller: none
    externalURL: https://harbor.malrusayni.com
    database:
      type: external
      external:
        host: harbor-pg-rw.harbor.svc
        port: 5432
        username: harbor
        password: ""  # set via valuesFrom
    redis:
      type: internal
    persistence:
      enabled: true
      persistentVolumeClaim:
        registry:
          size: 300Gi
    trivy:
      enabled: true
    core:
      resources:
        requests:
          memory: "256Mi"
          cpu: "100m"
        limits:
          memory: "512Mi"
          cpu: "500m"
    registry:
      resources:
        requests:
          memory: "256Mi"
          cpu: "100m"
        limits:
          memory: "512Mi"
          cpu: "500m"
  valuesFrom:
    - kind: Secret
      name: harbor-admin-secret
      valuesKey: HARBOR_ADMIN_PASSWORD
      targetPath: harborAdminPassword
    - kind: Secret
      name: harbor-pg-secret
      valuesKey: password
      targetPath: database.external.password
```

Write to `clusters/production/harbor/helm-release.yaml`.

```bash

git commit -m "feat(harbor): add HelmRepository + HelmRelease"
```

### Task 5: Ingress

**Files:**
- Create: `clusters/production/harbor/ingress.yaml`

- [ ] **Step 1: Create Ingress manifest**

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: harbor
  namespace: harbor
  annotations:
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
    kubernetes.io/ingress.class: "traefik"
spec:
  rules:
    - host: "harbor.malrusayni.com"
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: harbor-portal
                port:
                  number: 80
  tls:
    - hosts:
        - "harbor.malrusayni.com"
      secretName: harbor-tls
```

Write to `clusters/production/harbor/ingress.yaml`.

**Note:** Harbor's Helm chart creates two services: `harbor-portal` (UI on port 80) and `harbor-core` (API on port 80). The ingress points to `harbor-portal` which serves both the UI and proxies API requests. Harbor handles the routing internally.

## Pitfalls

1. **Secret sealing requires cluster access** — the user must run `kubeseal` against the live cluster. The plain Secret templates are gitignored references.
2. **HelmRelease valuesFrom** — `valuesKey` and `targetPath` syntax must match Flux's expected format. The `targetPath` uses dot-notation (e.g., `database.external.password`).
3. **Service name** — Harbor Helm chart creates `harbor-portal` Service (not `harbor`). The ingress must point to the correct service name.
4. **CNPG secret naming** — CloudNativePG expects the bootstrap secret name to match what's in `spec.bootstrap.initdb.secret.name`. The HelmRelease references the same secret for DB credentials.
5. **DNS** — `harbor.malrusayni.com` must resolve to the cluster's ingress IP before the TLS certificate can be issued.
